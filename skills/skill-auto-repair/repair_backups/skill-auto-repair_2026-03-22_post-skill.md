# Skill Auto-Repair

스킬이 작업 중에 깨졌을 때, 원인을 진단하고 **재발할 가능성이 있는 구조적 문제만** 스킬 파일에 패치한다. 일회성 오류는 스킬을 수정하지 않고 그냥 처리하고 작업을 이어간다. 사용자가 처음부터 다시 시작할 필요가 없도록 하는 것이 목표다.

---

## 언제 개입하는가

다음 상황에서 이 스킬이 자동으로 개입한다:

- 현재 실행 중인 스킬이 오류(예외, 스크립트 실패, 도구 호출 실패)를 냄
- 스킬 출력이 명백히 잘못됐거나 불완전함
- 스킬 지시사항이 존재하지 않는 파일, 도구, 경로를 참조함
- 스킬이 현재 입력의 엣지케이스를 처리하지 못하고 막힘
- Claude가 깨진 지시사항을 따르느라 같은 시도를 반복하고 있음

---

## Step 1: 어떤 스킬이 깨졌는지 파악

현재 컨텍스트에서 다음을 파악한다:

- **어떤 스킬**이 실행 중이었는가 (SKILL.md 경로 또는 스킬 이름)
- **무엇이 잘못됐는가** (오류 메시지, 잘못된 출력, 막힌 단계)
- **어느 시점**에서 실패했는가

스킬 이름을 특정하지 못하면 사용자에게 한 번만 묻는다: "어떤 스킬을 사용하다가 문제가 생겼나요?"

---

## Step 2: 화이트리스트 확인

`~/.claude/skills/skill-auto-repair/allowed_skills.json`을 읽는다.

깨진 스킬이 화이트리스트에 **없으면** 즉시 멈추고 알린다:

> "⚠️ `[스킬명]`은 자동 수정 허용 목록에 없어서 수정할 수 없어요.
> 허용하려면 `allowed_skills.json`에 추가해 주세요.
> 현재 허용된 스킬: [목록]"

화이트리스트에 **있으면** 다음 단계로 진행한다.

---

## Step 3: 스킬 파일 읽기 + 재발 가능성 판단

깨진 스킬의 관련 파일을 읽은 뒤, **이 오류가 다시 발생할 가능성이 있는가**를 판단한다. 이것이 스킬을 수정할지 말지를 결정하는 핵심이다.

### 스킬을 수정해야 하는 경우 (구조적 문제)

스킬의 지시사항 자체에 결함이 있어서, 같은 스킬을 다시 써도 같은 문제가 반복될 것이 예상될 때:

- 스킬이 흔히 발생하는 입력 유형을 처리하는 방법을 명시하지 않음
- 파일 경로, 도구 이름, API가 잘못 참조되어 있음 (항상 실패함)
- 지시사항이 구조적으로 모호해서 Claude가 매번 잘못 해석할 가능성이 높음
- 스크립트에 로직 버그가 있음
- 필수 선행 단계가 누락되어 있음

### 스킬을 수정하지 않는 경우 (일회성 문제)

오류가 이번 상황에 특수한 것이어서, 스킬을 바꿔도 다음번에는 문제가 없을 것이 예상될 때:

- 네트워크 타임아웃, 임시 파일 소실 등 환경적 오류
- 사용자가 이번에만 제공한 비정상적인 입력 형식
- 외부 서비스의 일시적 장애
- 이 세션/프로젝트에만 해당하는 특수한 사정
- 랜덤하게 발생하는 런타임 오류 (재현 불가능할 가능성 높음)

판단이 어려운 경우, 같은 스킬을 같은 방식으로 쓰는 다른 사람도 이 오류를 겪을 것인지 생각해본다. 그렇다면 수정, 아니라면 일회성으로 처리한다.

---

## Step 4A: 구조적 문제 — 백업 후 스킬 파일 수정

Step 3에서 구조적 문제로 판단됐을 때만 이 단계를 실행한다.

### 수정 전: 반드시 백업부터

파일을 건드리기 전에 현재 SKILL.md를 백업한다:

```
~/.claude/skills/skill-auto-repair/repair_backups/
└── [스킬명]_YYYY-MM-DD_HH-MM.md
```

백업 파일은 SKILL.md 하나만 저장한다 (스크립트/참조 파일은 git으로 추적 가능하므로 제외). 백업이 완료된 후에만 원본을 수정한다.

사용자가 "되돌려줘"라고 하면 가장 최근 백업 파일로 복원한다.

### 수정 원칙

- **최소 변경**: 깨진 것만 고친다. 작동하는 부분은 건드리지 않는다
- **즉시 적용**: 확인 없이 파일에 바로 반영한다
- **범위 제한**: 스킬의 핵심 목적이나 전체 구조를 바꾸지 않는다

### 흔한 수정 유형

| 원인 | 수정 방법 |
|------|-----------|
| 누락된 엣지케이스 | 해당 케이스 처리 로직을 SKILL.md에 추가 |
| 깨진 파일/도구 참조 | 경로 수정 또는 현재 사용 가능한 대안으로 교체 |
| 모호한 지시사항 | 더 명확하게 재작성 |
| 스크립트 버그 | 해당 스크립트 직접 수정 |
| 누락된 선행 단계 | 단계 추가 |

### 수정 후: 이력 기록

`~/.claude/skills/skill-auto-repair/repair_log.md`에 한 줄 추가한다:

```
| YYYY-MM-DD HH:MM | [스킬명] | [수정 내용 한 줄] | 구조적 수정 |
```

파일이 없으면 헤더와 함께 새로 만든다:
```
| 날짜 | 스킬 | 수정 내용 | 유형 |
|------|------|-----------|------|
```

### 수정 후: Git 커밋 & 푸시

이력 기록 직후 아래 명령을 실행한다:

```bash
cd ~/.claude && git add skills/[스킬명]/ && git commit -m "fix: [스킬명]/SKILL.md — [수정 내용 한 줄]" && git push
```

- 커밋 메시지는 실제 수정 내용을 한 줄로 요약한다
- push 실패 시 커밋은 유지하고, 사용자에게 별도 알리지 않는다 (작업 흐름을 방해하지 않도록)

수정 후 알린다:

> "🔧 스킬 수정: [무엇을 왜 바꿨는지 한 줄]. 작업을 이어서 진행할게요."

---

## Step 4B: 일회성 문제 — 수정 없이 처리

Step 3에서 일회성으로 판단됐을 때는 스킬 파일을 건드리지 않는다. 그냥 이번 오류를 직접 해결하고 작업을 계속한다. 사용자에게는 한 줄로만 알린다:

> "⚡ 일회성 문제로 판단해서 스킬은 수정하지 않았어요. 작업을 이어서 진행할게요."

---

## Step 5: 원래 작업 재개

수정 여부와 관계없이, 사용자가 다시 요청할 필요 없이 중단된 지점부터 바로 작업을 계속한다.

---

## 화이트리스트 관리

허용 목록은 `~/.claude/skills/skill-auto-repair/allowed_skills.json`에 저장된다.

```json
{
  "allowed_skills": [
    "스킬-디렉토리명-1",
    "스킬-디렉토리명-2"
  ],
  "note": "수정을 허용할 스킬의 디렉토리명을 여기에 추가하세요 (예: skill-creator, unity-mcp-skill)"
}
```

사용자가 스킬을 추가/제거하도록 요청하면 이 파일을 직접 수정한다.

**스킬 디렉토리명** 기준이다 — `~/.claude/skills/` 아래의 폴더 이름.

---

## Step 6: 작업 완료 후 정리 (선택적)

원래 작업이 완전히 끝난 뒤, 이번 작업을 통해 스킬에서 발견된 비효율이 있으면 조용히 정리한다. 오류가 없었더라도 실행 중에 눈에 띈 군더더기가 있으면 이 단계에서 처리한다.

### 정리할 가치가 있는 것

- **중복 지시사항**: 같은 내용을 두 군데서 말하고 있는 경우
- **상충하는 지시사항**: 서로 모순되는 두 규칙이 공존하는 경우
- **죽은 참조**: 더 이상 존재하지 않는 파일/스크립트를 가리키는 링크
- **사용되지 않는 스크립트**: `scripts/`에 있지만 SKILL.md에서 언급조차 안 되는 파일
- **불필요하게 복잡한 단계**: 더 간단하게 표현할 수 있는 지시사항

### 정리하지 말아야 할 것

- 스타일이 마음에 안 드는 부분 (취향 차이)
- 지금 당장 쓰이지 않지만 의도적으로 남겨둔 것처럼 보이는 부분
- 잘 작동하는 지시사항 (더 나은 방법이 있더라도 굳이 바꾸지 않음)

정리 후 `repair_log.md`에 한 줄 기록하고, git 커밋 & 푸시한다:

```
| YYYY-MM-DD HH:MM | [스킬명] | [정리 내용 한 줄] | 정리 |
```

```bash
cd ~/.claude && git add skills/[스킬명]/ && git commit -m "chore: [스킬명]/SKILL.md — [정리 내용 한 줄]" && git push
```

> "🧹 정리: [무엇을 정리했는지 한 줄]."

정리할 것이 없으면 아무 말도 하지 않는다.

### 백업 파일 정리

`repair_backups/`에 같은 스킬의 백업이 5개를 넘으면 가장 오래된 것부터 삭제해서 5개만 유지한다. 로그는 계속 쌓아도 되지만 백업 파일은 용량을 차지하므로 관리한다.

---

## 하지 말아야 할 것

- 화이트리스트에 없는 스킬 수정
- 재발 가능성 판단 없이 모든 오류에 스킬을 수정하기
- 작업 중에 정리하기 — 정리는 반드시 작업이 끝난 후에
- "개선"이라는 명목으로 잘 작동하는 부분 건드리기
- 사용자가 요청하지 않은 기능 추가
- 스킬의 핵심 목적 변경
- 수정 때문에 원래 작업을 잊어버리기 — 반드시 재개할 것
